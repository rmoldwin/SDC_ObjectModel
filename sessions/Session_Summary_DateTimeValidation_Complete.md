## Scope Delivered

### Soft-reject validation for all 12 date / date-part types
- `date`, `dateTime`, `dateTimeStamp`, `time`, `gDay`, `gMonth`, `gMonthDay`, `gYear`, `gYearMonth`, `duration`, `dayTimeDuration`, `yearMonthDuration`.
- Invalid values are **never assigned**: the setter / deserialization path keeps the previous value (or leaves the field unset), does not throw, fires the validation event, and records the offending value in `BaseType.RejectedValues` with a message that **includes the offending value**.
- Stray time components on date-only types (and vice versa) are hard soft-rejected per the recorded decision.

### Exact 4-serializer round-trip parity (Deviation 1 — resolved)
- **Root cause:** only **BSON** shifted DateTimes by the host UTC offset (the earlier "BSON/MsgPack" note was wrong — MsgPack was already exact). The BSON *write* side correctly stored an absolute UTC instant; the *read* side re-projected it to host-local because `BsonDataReader.DateTimeKindHandling` defaults to `DateTimeKind.Local`.
- **Fix:** pin `bsonDataReader.DateTimeKindHandling = DateTimeKind.Utc` on both BSON read paths (`DeserializeBson`, `DeserializeBsonValidating`) in `SdcSerializerBson.cs`. This is a strict, model-wide correctness fix for **every** DateTime read through BSON — the stored instant now returns unchanged.
- **Residual (by design, not a defect):** XML preserves `DateTimeKind.Unspecified`; JSON/BSON/MsgPack normalize to `Utc`. That is a Kind *label* difference only — `DateTime` equality compares the absolute instant, which is now identical in all four. The XSD value is the lexical instant, not the Kind.

### Issue #10 — gMonth / gYearMonth / dayTimeDuration tree-builder bindings (Deviation 2 — resolved)
- **Root cause (NOT a regex / prefix-ambiguity issue):** wrong `[XmlElement]` bindings in the generated `DataTypes_DEType.cs`. `[XmlElement("gYearMonth", typeof(gMonth_DEtype))]` bound the wrong CLR type (so `gMonth_DEtype` matched two element names → ambiguous, and `gYearMonth_DEtype` matched zero), and there was **no** `[XmlElement("dayTimeDuration", …)]` binding at all.
- **Fix (Option 1, user-approved):** hand-corrected the bindings (`gYearMonth` → `typeof(gYearMonth_DEtype)`; added the missing `dayTimeDuration` binding), each annotated with an `ISSUE #10 HAND-CORRECTION` block. Suspected original cause: an earlier .NET / xsd2code++ collapsing structurally-similar CLR types.
- **Regen survival:** `git mv`'d the file out of the regen-special `SDC Unmodified Classes/` into `SDC Customized Classes/Hand-Corrected Generated Classes/` (history preserved), replaced the `<auto-generated>` header with a HAND-CORRECTED provenance block, and added a `README.md` documenting that the `partial class` makes any regenerated duplicate a loud build failure — re-apply the marked corrections and delete the regenerated copy.
- Removed the non-public deserialization-ctor workaround from the boundary tests; all twelve types now build and exact-round-trip through the **standard tree API**.

### Supporting infrastructure
