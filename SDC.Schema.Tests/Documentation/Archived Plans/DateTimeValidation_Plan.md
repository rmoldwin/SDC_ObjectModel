# Plan вАФ Soft-Reject Validation & Helpful Messaging for XML Date / Date-Part Types

**Status:** Plan only вАФ implementation begins in a new session. START IN PLAN MODE; do not implement
until the plan is reviewed and approved.

**Builds on:** Issue #8 soft-reject infrastructure (commit `f7fd14a`, merged to
`Features/NET10/Net10Main`). That work gave us `SdcUtil.ValidateAndRaise(value, ctx) вЖТ bool`,
`BaseType.RejectedValues` / `HasRejectedValues` / `ClearRejectedValue(s)`, the `SdcRejectedValue`
record, the `SuppressValidation`-gated event/collector pipeline, and the regen-safe post-processor
`Apply_SoftReject_Setters.py`. **This plan reuses all of it** вАФ every date type must land its invalid
values in `RejectedValues`, never in the typed field, never throwing.

**Authoritative goal (from the user):** The SDC OM must NEVER store an invalid date/date-part value вАФ
set programmatically or during deserialization of an invalid source document (XML/JSON/BSON/MsgPack).
On a value that fails validation the path must (1) keep the prior/unset value, (2) NOT throw
(soft reject), (3) fire `SdcValidationEvents.ValidationOccurred` + populate `ValidationCollector` when
active, recording the offending value in `RejectedValues`. **Additionally for this work:** the error
messages must be *exceptionally* helpful вАФ they must teach the user the correct XML lexical form,
because these date/date-part types are unfamiliar and easy to get wrong. Each message must mirror the
official XML Schema explanation and name the specific offending portion.

### 1.3 Regression to fix first вАФ `dateTimeStamp` is broken by issue #8

`dateTimeStamp_Stype.val` and the four `dateTimeStamp_DEtype` facets are `System.DateTime` but carry
`[RegularExpression(".*(Z|(\+|-)dd:dd)")]`. Under #8 soft-reject, `ValidateAndRaise` runs
`Validator.TryValidateProperty`, which calls `RegularExpressionAttribute.IsValid` on the **DateTime's
`.ToString()`** (e.g. `"6/22/2026 11:46:03 PM"`). A bare `System.DateTime` has no timezone, so the
pattern can **never** match вЖТ **every dateTimeStamp value is now soft-rejected and dropped.** This is
the unresolved Open-Question I-1 in `Session_Handoff_ValidationPlan.md`, now upgraded from "false event"
to "hard regression." Must be fixed as the first task of this work.

### 1.4 Current error mechanism is weak

`IDataHelpers.AddDataTypesDE` validates the *lexical string* with correct XSD regexes but reports via
`StoreError("Supplied value parameter could not be parsed as <type>")` вАФ generic, omits the offending
value, omits the correct form, and is decoupled from the `RejectedValues` store. The XSD regexes there
are the **authoritative source of truth** and should be promoted to shared constants and reused
(don't re-derive them).

## 2. Core design problem & decision

Two fundamentally different shapes:

**A. String-backed types** (`gYear`, `gYearMonth`, `gMonth`, `gMonthDay`, `gDay`, `duration`,
`dayTimeDuration`, `yearMonthDuration`). The `val` *is* the XSD lexical string, so a `[RegularExpression]`
facet with the canonical XSD pattern flows naturally through the existing #8 `ValidateAndRaise`
pipeline. Setting `val = "garbage"` will soft-reject and record. **This is the easy, fully-aligned path.**

**B. DateTime-backed types** (`date`, `dateTime`, `time`, `dateTimeStamp`). The typed field
(`System.DateTime`) *cannot represent* an invalid XSD lexical value вАФ by the time you have a `DateTime`,
the offending string is gone, and many XSD constraints don't even map (separate timezone offset, year 0
/ negative years / year > 9999, `24:00:00`, a `date` that wrongly carries a time, a `time` that wrongly
carries a date). **The lexical validation must happen at the string boundary** (the `AddDataTypesDE`
parse path and any future string setters), producing a soft-reject + `RejectedValues` entry keyed to the
node, while the `DateTime val` setter itself only guards the few things a `DateTime` can express. The
separate `timeZone` string gets its own custom validator (offset range вИТ14:00вА¶+14:00).

**Recommended decision (to confirm with user):**
- **String-backed:** add the canonical XSD `[RegularExpression]` facet (regen-safe via `[MetadataType]`
  buddy classes вАФ Section 3) so they ride the existing #8 setter pipeline. Replace the weak/partial
  duration regexes with full ISO-8601 patterns.
- **DateTime-backed:** introduce a custom `XsdDateTimeLexicalAttribute(XsdDateKind kind)` validation
  attribute that validates a **lexical string** (used wherever the raw string is available вАФ the parse
  path and a new `SetLexicalValue(string)` entry point), plus a `XsdTimezoneOffsetAttribute` on the
  `timeZone` property. The `DateTime val` setter keeps a light guard (range/representability). Route the
  parse path's failures through `SdcUtil.RecordRejectedValue` + the event hub instead of `StoreError`'s
  generic text, so invalid source documents surface every error and leave fields unset.
- **`dateTimeStamp`:** replace the broken `[RegularExpression]` on the DateTime property with the
  timezone-required lexical validation at the string boundary; remove the impossible regex (resolves I-1).

## 3. Regeneration-safe implementation strategy

Generated date files must not be hand-edited (repo policy). Options, in priority order:

1. **`[MetadataTypeAttribute]` buddy classes** for string-backed `val` facets. DataAnnotations honors
   metadata "buddy" types: declare `partial class gYear_Stype` (regen-safe, in a custom folder) decorated
   with `[MetadataType(typeof(GYearMeta))]`, where `GYearMeta` carries `[RegularExpression(pattern,
   ErrorMessage=...)]` on a matching `val` member. **Verify** that the #8 `Validator.TryValidateProperty`
   path (which uses `ValidationContext(this,вА¶)` + `MemberName`) actually resolves buddy-class metadata вАФ
   `TryValidateProperty` historically does NOT consult `[MetadataType]` unless
   `TypeDescriptor.AddProviderTransparent`/`AssociatedMetadataTypeTypeDescriptionProvider` is registered.
   **Spike this first.** If buddy metadata is not honored, fall back to option 2.
2. **Custom validation invoked explicitly by our own helper.** Since *we* own `SdcUtil.ValidateAndRaise`,
   add an optional per-(type,member) rule lookup there (e.g. a static registry mapping
   `(Type, memberName) вЖТ ValidationAttribute[]`) populated from a custom-code table. This keeps all date
   rules in editable files and needs no generated-file edits and no buddy-metadata reflection quirk.
   **This is the most robust option and the recommended primary approach.**
3. **`partial` + post-gen script** (like `Apply_SoftReject_Setters.py`) only if 1вАУ2 are insufficient. If
   any generated date file must change, extend the existing post-processor and **stop to get explicit
   user approval** (list exact files) per repo policy.

The custom rule registry (option 2) also cleanly solves the DateTime-backed case: the lexical attribute
and timezone attribute live in code, and the parse path calls the same helper, so messages and recording
are identical everywhere.

**New/edited files will live in editable folders only:**
`SDC.Schema/SDC.Schema/SDC Customized Classes/` (validation attributes, buddy/partial metadata,
divergence remarks), `SDC.Schema/Utility Classes/` (`SdcUtil` registry hook, shared XSD pattern
constants), `SDC.Schema/Interfaces/IDataHelpers.cs` (route parse errors through the new pipeline).

## 4. XSD lexical rules, .NET divergences, and message templates

Promote the regexes already in `IDataHelpers` to `public const string` fields on a new
`XsdDateTimePatterns` class (single source of truth, reused by attributes, the parse path, and tests).

| Type | Canonical XSD lexical form | Notable .NET divergence to document |
|---|---|---|
| `xs:dateTime` | `[-]CCYY-MM-DDThh:mm:ss[.fff][Z\|¬±hh:mm]` | `System.DateTime` range 0001вАУ9999 (XSD has no upper bound); no year 0 (both agree); offset stored separately; `24:00:00` allowed by XSD, not by `DateTime`; ticks=100 ns truncates finer fractions |
| `xs:date` | `[-]CCYY-MM-DD[Z\|¬±hh:mm]` | A lexical time component must be **rejected**, not silently truncated by `.Date` |
| `xs:time` | `hh:mm:ss[.fff][Z\|¬±hh:mm]` | A lexical date component must be **rejected**; `DateTime` injects a placeholder date |
| `xs:dateTimeStamp` | `xs:dateTime` **with required** `[Z\|¬±hh:mm]` | timezone mandatory; current regex on DateTime is impossible (I-1) |
| `xs:gYearMonth` | `[-]CCYY-MM[Z\|¬±hh:mm]` | string-backed; no `DateTime` constraint |
| `xs:gYear` | `[-]CCYY[Z\|¬±hh:mm]` (вЙ•4 digits) | string-backed |
| `xs:gMonthDay` | `--MM-DD[Z\|¬±hh:mm]` | string-backed; MM 01вАУ12, DD valid for month |
| `xs:gDay` | `---DD[Z\|¬±hh:mm]` | string-backed; DD 01вАУ31 |
| `xs:gMonth` | `--MM[Z\|¬±hh:mm]` | string-backed; MM 01вАУ12 |
| `xs:duration` | `[-]P[nY][nM][nD][T[nH][nM][nS]]`, вЙ•1 component, `T` only if a time part follows | partial/incomplete current regex |
| `xs:dayTimeDuration` | `xs:duration` restricted to `D,T,H,M(time),S` (no `Y`, no date-`M`) | current `[^YM]*[DT].*` too loose |
| `xs:yearMonthDuration` | `xs:duration` restricted to `Y,M(date)` only | current `[^DT]*` too loose |

Timezone offset facet (shared): `Z` or `¬±hh:mm` with the offset in **вИТ14:00 вА¶ +14:00** (XSD rule);
applies to every type's optional zone and to the `timeZone` property.

### 4.1 Message style (mandatory)

Each message must: (a) quote the offending value, (b) name the XSD type, (c) give the canonical form with
a concrete valid example, and (d) pinpoint the specific violation. Template:

> `'{value}' is not a valid xs:{type}. The correct form is {canonicalForm} (for example, {example}).
> {specificProblem}`

Worked examples to implement & test:
- `'2026-13-05'` вЖТ *"вА¶not a valid xs:date. The correct form is CCYY-MM-DD with an optional timezone
  (for example, 2026-06-22 or 2026-06-22Z). The month must be 01вАУ12; you supplied '13'."*
- `'2026-06-22T10:00:00'` set as `dateTimeStamp` вЖТ *"вА¶not a valid xs:dateTimeStamp. A dateTimeStamp is a
  dateTime that REQUIRES a timezone, e.g. 2026-06-22T10:00:00Z or 2026-06-22T10:00:00-05:00. Your value
  has no timezone."*
- `'--13'` as `gMonth` вЖТ *"вА¶not a valid xs:gMonth. The correct form is --MM with an optional timezone
  (for example, --06). The month must be 01вАУ12; you supplied '13'."*
- `'P'` as `duration` вЖТ *"вА¶not a valid xs:duration. The correct form is PnYnMnDTnHnMnS with at least one
  component, e.g. P3Y6M4DT12H30M5S or PT15M. An empty duration 'P' is not allowed."*
- `'PT1Y'` as `dayTimeDuration` вЖТ *"вА¶not a valid xs:dayTimeDuration. This type allows only day and time
  parts (D, H, M, S after T), e.g. P5DT12H. Year/Month parts are not permitted; you supplied 'Y'."*
- `'+15:00'` zone вЖТ *"вА¶not a valid timezone. Offsets must be Z or ¬±hh:mm within вИТ14:00вА¶+14:00; you
  supplied '+15:00'."*

## 8. Open questions вАФ user decisions

> **Branch:** this work proceeds in a new session on `Features/NET10/Net10_DateTimeValidation`, branched
> off `Features/NET10/Net10Main` (which contains issue #8 at `f7fd14a`/merged).

1. **Buddy metadata vs rule registry** (task 1 spike) вАФ confirm the regen-safe mechanism. **DECIDED:
   use the rule registry in `SdcUtil`.** Rationale: `Validator.TryValidateProperty` (called by
   `ValidateAndRaise`) does not honor `[MetadataType]` buddy attributes without registering an
   `AssociatedMetadataTypeTypeDescriptionProvider`, and even then only on `TypeDescriptor`-aware paths вАФ
   an unproven runtime risk. A custom `(Type, memberName) вЖТ ValidationAttribute[]` registry consulted
   inside `ValidateAndRaise` keeps every date rule in an editable file, needs zero generated-file edits,
   survives regen automatically, and is deterministic + unit-testable. (Buddy classes remain a fallback
   only if a spike later proves them reliable.)
2. **`date`/`time` lexical strictness** вАФ **DECIDED: hard soft-reject.** Setting an `xs:date` from a
   string carrying a time component (or an `xs:time` carrying a date component) is rejected with a
   helpful message rather than silently truncated via `.Date`. Rationale: matches the "never store an
   invalid value" contract and mirrors how real-world UI date/time pickers separate the components.
3. **`timeZone` representation** вАФ keep as validated `string`, or migrate to a `TimeSpan`/offset type?
   (`IDataHelpers` has a TODO musing this.) Recommendation: keep string + `XsdTimezoneOffsetAttribute`
   for this work; defer any type migration.
4. **Scope of programmatic entry points** вАФ do we add a `SetLexicalValue(string)` API on DateTime-backed
   types so callers can set+validate a raw XSD string directly, or is the `AddDataTypesDE` path
   sufficient? Recommendation: add the lexical entry point for parity and testability.
