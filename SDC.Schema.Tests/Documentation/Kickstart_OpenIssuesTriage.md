## Open issues to triage

### #12 — anyURI regex uses XSD char-class syntax invalid in .NET — **HIGH (bug)**
- **What:** the `anyURI` case in `IDataHelpers.AddDataTypesDE` builds a `Regex` containing XSD codepoint ranges (`#x1-#xD7FF`, `#xE000-#xFFFD`, `#x10000-#x10FFFF`). `#x...` is **not** a valid .NET regex escape → `new Regex(...)` throws `ArgumentException` the moment that branch executes. Latent today only because no test exercises an `anyURI` data element.
- **Location:** `SDC.Schema/SDC.Schema/Utility Classes/SdcUtil.cs` (wherever `IDataHelpers.AddDataTypesDE` lives), the `anyURI` case.
- **Suggested fix:** translate the ranges to .NET syntax (`#x1`→`\u0001`/`\^A`, `#xD7FF`→`\uD7FF`, plus surrogate handling), **or** replace the hand-rolled pattern with the canonical `Uri.TryCreate(value, UriKind.RelativeOrAbsolute, out _)`.
- **Notes:** pre-existing, out of date-validation scope (was tracked as I-2). Likely the best first pick — highest severity, well-scoped, add a test that actually hits the branch.

### #9 — migrate date/time `timeZone` from validated string to TimeSpan/offset — **enhancement**
- **What:** the date-validation work intentionally kept `timeZone` on `date_Stype` / `dateTime_Stype` / `time_Stype` as a **validated string** (`XsdTimezoneOffsetAttribute`, accepts XSD `Z`/`±hh:mm` and the .NET `±h:mm:ss` form `AddDataTypesDE` emits). Migrate to a strongly-typed offset (`System.TimeSpan` or dedicated type; also evaluate moving `val` to `DateTimeOffset`).
- **Acceptance:** regen-safe (registry/partials/extensions, no `<auto-generated>` edits); preserve XML/JSON/BSON/MsgPack round-trip including legacy `±h:mm:ss` documents (back-compat read path); update `XsdTimezoneOffsetAttribute`/`XsdDateTimePatterns` + the companion doc; tests mirroring `DateResponseTypeBoundaryTests`/`DateResponseTypeRoundTripTests`.
- **Refs:** `DateTimeValidation_XSD_vs_NET.md` §2.1 (timeZone) + §2.4 (BSON/MsgPack DateTime), `XsdDateTimePatterns.cs` (`IsValidTimezoneOffset`).
