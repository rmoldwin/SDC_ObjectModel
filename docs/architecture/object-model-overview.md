# Object Model Overview

This chapter introduces the SDC.Schema Object Model (OM) — the in-memory class hierarchy that
represents a Structured Data Capture (SDC) form (a Form Design File, or FDF, once loaded). It is
the foundation every other chapter builds on: (de)serialization
([serialization.md](serialization.md)), validation ([validation.md](validation.md)), the rules
engine ([rules.md](rules.md)), tree comparison ([compare-trees.md](compare-trees.md)), and the
`SdcUtil` helper library ([sdc-util.md](sdc-util.md)) all operate on this tree.

## Where the code lives

The `SDC.Schema` project is generated substantially by Xsd2Code++ (https://www.xsd2code.com) from
the SDC XML Schema Definition (XSD) files, then hand-customized. Its `Schema Classes/` folder is
split into four subfolders, each with a different editing policy:

- **`SDC Schema Files/`** — the Xsd2Code++ output as originally generated from the XSD, including
  an `Excluded SDC Schema Classes/` subfolder for generated classes that aren't compiled into the
  build. Treated as read-only generator output.
- **`SDC Unmodified Classes/`** — generated classes left as-is (no hand customization needed).
- **`SDC Constructor Removed/`** — generated classes where the constructor was stripped out (see
  below) so a hand-written replacement can be layered on elsewhere.
- **`SDC Customized Classes/`** — the generated classes that received hand modifications (for
  example, `BaseType.cs`) directly in the generated file.

All four folders begin with an `<auto-generated>` comment banner. Per the project's Auto-Generated
File Policy, files carrying that banner must not be edited without explicit sign-off, regardless
of which of these folders they're in — custom behavior is added instead via `partial` classes
(consolidated in `Partial Classes/PartialClasses.cs`), extension methods (`Extensions/`), or
derived/computed properties (`SDC Customized Classes/`). `Utility Classes/` and `Interfaces/` hold
hand-written, freely-editable helper code (including `SdcUtil.cs` and `CompareTrees.cs`).

## The node hierarchy: `BaseType`

Every SDC element in the tree is represented by a class deriving, directly or indirectly, from
`BaseType`. `BaseType` itself is not tied to one SDC element — it is inherited by roughly 150
generated SDC complex types (enforced via a large block of `[XmlInclude(typeof(...))]` attributes
that also drives XML polymorphic serialization, see [serialization.md](serialization.md)). Every
node in the tree — whatever its concrete SDC type — exposes the same core navigation and identity
surface, defined on the `IBaseType` interface (`Interfaces/IBaseType.cs`):

- **`ObjectID`** (`int`) — a monotonically increasing integer assigned as each node is constructed
  during a single build/deserialization pass. It is **not stable across a serialize/deserialize
  round trip** — a freshly deserialized tree gets new `ObjectID` values — so it must not be used
  as a durable, persisted identifier.
- **`ObjectGUID`** (`Guid`) — a unique identifier assigned per node at construction time; `sGuid`
  (a short GUID form), if used, is derived from it.
- **`ParentNode`** (`BaseType?`) — the immediate parent node, or `null` for the tree's top node.
  `ParentID` is a convenience wrapper returning `ParentNode.ObjectID` (or `-1` if there is none).
- **`ParentIETnode`** / **`ParentIETnodeID`** — the closest ancestor that is an
  `IdentifiedExtensionType` (IET, see below), and its ID.
- **`TopNode`** (`ITopNode?`) — the root of the tree this node belongs to.
- **`NodeType`** (`ItemTypeEnum`) — the SDC type of the current node.
- **`ElementName`** / **`ElementPrefix`** / **`BaseName`** — inputs to how the node's XML element
  name (the `@name` attribute) is generated or customized.

### `IdentifiedExtensionType` (IET)

Many SDC complex types derive (directly or through intermediate types) from
`IdentifiedExtensionType`, commonly abbreviated **IET** throughout this codebase's docs, comments,
and Best Practice (`BP-*`) rule IDs (see [qa-best-practices.md](qa-best-practices.md)). An IET
node carries a durable, form-author-assigned unique ID (distinct from the transient `ObjectID`
above) — this is the identity that survives a round trip and that rules, validation, and
`CompareTrees` ([compare-trees.md](compare-trees.md)) use to recognize "the same logical node"
across two different trees or two versions of the same form.

### Tree construction and mutation

Nodes are added to a tree either by deserializing a complete SDC document (see
[serialization.md](serialization.md)) or by direct, in-memory construction/attachment through the
public parameterized constructors (every SDC node class has one, requiring a non-null
`parentNode`) and the move/attach/remove helpers described in `IMoveRemove`/`INavigate`
(implemented by `IBaseType`). `SdcUtil.cs` ([sdc-util.md](sdc-util.md)) provides the bulk of the
tree-walking, lookup, and caching helpers used both internally and by consumers of the OM.

### Thread safety and tree stability

Building, mutating, and concurrently reading a tree has been the subject of two related, but
distinct, investigations — a completed desktop/single-process one and a separate, still-open one
under real WebAssembly (WASM) multi-threading. See
[thread-safety.md](thread-safety.md) (read its scope-caveat banner first) and
[tree-stability.md](tree-stability.md) for full detail before writing code that mutates a tree
from more than one thread.
