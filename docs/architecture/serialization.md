# Serialization

> **Status:** Living document. The bug-history and generator-architecture sections below were
> consolidated from `BsonJsonSerializationBugReport.md` (originally in
> `SDC.Schema.Tests/Documentation`, now archived — see [../changes/](../changes/)).

This chapter covers the SDC.Schema serializer architecture across all four supported wire
formats, all four generated by Xsd2Code++ (https://www.xsd2code.com) and then hand-modified for
the SDC Object Model (OM)'s deep polymorphic inheritance:

- **XML** — `SdcSerializer<T>`, built on `System.Xml.Serialization.XmlSerializer`, handling
  polymorphism via roughly 150 `[XmlInclude]` attributes on `BaseType`.
- **JSON** — `SdcSerializerJson<T>`, built on `Newtonsoft.Json`'s `JsonConvert`, using
  `TypeNameHandling.All` and `ConstructorHandling.AllowNonPublicDefaultConstructor` for
  polymorphic round-trips (diverging from stock generated JSON serializers, which use an empty
  `new JsonSerializerSettings()`).
- **BSON (Binary JSON)** — `SdcSerializerBson<T>`, using `Newtonsoft.Json.Bson`'s
  `BsonDataWriter`/`BsonDataReader` with a `Newtonsoft.Json.JsonSerializer` (settings align with JSON).
  BSON bytes are stored as Base64 strings.
- **MessagePack (MsgPack)** — `SdcSerializerMsgPack<T>`, using the `Newtonsoft.Msgpack` v0.1.11
  NuGet package (a wrapper around `MsgPack.Cli`/`MsgPack.Serialization`), calling
  `MessagePackSerializer<T>.Pack`/`Unpack`. Returns `byte[]` (not Base64). Public APIs:
  `SerializeMsgPack`, `DeserializeMsgPack`, `SaveToFileMsgPack`, `LoadFromFileMsgPack`.
- Ad hoc/"any" attribute and namespace round-tripping across all of the above formats, including
  multiple mixed namespaces and inherited/default namespace usage (see
  [qa-best-practices.md](qa-best-practices.md) for the relevant numbered example and QA rules).

## Why BSON is driven by `JsonSerializer`

`Newtonsoft.Json.Bson` provides no independent serializer class. `BsonDataWriter`/`BsonDataReader`
are subclasses of `JsonWriter`/`JsonReader` — a binary wire-format translation layer over the same
Newtonsoft JSON object model. `JsonSerializer` drives BSON exactly as it drives text JSON, and all
settings (`TypeNameHandling`, `ConstructorHandling`, etc.) apply identically. This is the only API
`Newtonsoft.Json.Bson` provides, and it is the pattern the Xsd2Code++ generator uses. BSON output
is stored as a **Base64 string** (`Convert.ToBase64String`/`FromBase64String`).

## Required settings and why (bug history)

The **stock** Xsd2Code++-generated BSON/JSON serializers use no special settings
(`new JsonSerializer()` / empty `new JsonSerializerSettings()`), which is correct for simple,
flat, non-polymorphic schemas but **insufficient for the SDC OM**, whose deep polymorphism
(e.g. `ChildItemsType.ChildItemsList` is declared `List<IdentifiedExtensionType>` but holds
concrete subtypes like `SectionItemType`/`QuestionItemType`/`DisplayedType` at runtime) and
protected/internal deserialization constructors require two settings the generator does not add
by default. Historically, both were missing — this was a real, high-severity bug (both formats
silently failed to restore a valid tree on deserialize) that has since been fixed:

| Setting | Applies to | Symptom when missing | Why it's needed |
|---|---|---|---|
| `ConstructorHandling.AllowNonPublicDefaultConstructor` | JSON + BSON | `NullReferenceException`: "parentNode can only be null if this object implements ITopNode" | Every SDC class has a **public** constructor requiring a non-null `parentNode`, plus a **protected/internal parameterless** constructor for deserialization only. Without this setting, Newtonsoft picks the public constructor and passes `null`. |
| `TypeNameHandling.All` | JSON + BSON | `JsonSerializationException`: "Could not create an instance of type SDC.Schema.IdentifiedExtensionType. Type is an interface or abstract class..." (or a silently wrong concrete type, losing subtype-specific data) | Without `"$type"` discriminators, Newtonsoft can only instantiate the *declared* (often abstract) property type, not the real runtime subtype. |

Both are now applied consistently in `SdcSerializerBson<T>` (a single cached `JsonSerializer`
instance) and `SdcSerializerJson<T>` (`JsonSerializerSettings` constructed per call). A BSON→JSON
intermediate conversion path (`JToken.ReadFrom(new BsonDataReader(...))`) was also affected by the
missing `TypeNameHandling.All` and is fixed as a side-effect of the same change.

> **Security note:** `TypeNameHandling.All` is safe for internal/trusted round-trips, but is a
> known deserialization gadget-chain risk when accepting JSON/BSON from **untrusted external**
> sources. If SDC documents are ever accepted from outside this codebase's control, supply a
> custom `SerializationBinder` that whitelists only `SDC.Schema` assembly types.

## MessagePack (MsgPack) status

Per Xsd2Code++'s documented pattern, `DefaultSerialiser = MessagePackSerializer` generates a
`MessagePackSerializer<T>.Pack`/`Unpack`-based serializer with no special settings analogous to
`TypeNameHandling`/`ConstructorHandling` — it relies on native CLR reflection and does **not**
understand `[XmlInclude]` or `[JsonProperty]` attributes. An earlier workaround replaced
`Pack`/`Unpack` with an XML-bytes tunnel (`Encoding.UTF8.GetBytes(SdcSerializer<T>.Serialize(obj))`)
so round-trip tests passed while `MsgPack.Cli`/`Newtonsoft.Msgpack` was actually bypassed
entirely. **This workaround has been reverted** — `SdcSerializerMsgPack<T>` now calls the real
`Pack`/`Unpack`, which is the correct generated pattern, but means polymorphic SDC types may not
currently round-trip correctly through MsgPack. `MsgPackRoundTripFidelityTest` and
`CloneSdcSubtreeMpackTest` failing is therefore **expected, documented behavior**, not a
regression — the failure must stay visible until MsgPack support is properly fixed (e.g. custom
`IMessagePackSerializer` adapters) or the library is replaced. Tracked in [roadmap.md](roadmap.md).

## Common structural pattern across all four serializers

1. A single lazily-initialized static serializer instance (BSON caches a `JsonSerializer`; JSON
   builds `JsonSerializerSettings` inline per call).
2. Three `Deserialize` overloads per format: a throwing core method, a `(out T, out Exception)`
   try-pattern, and a `(out T)` convenience overload that discards the exception.
3. `SaveToFileXxx`/`LoadFromFileXxx` helpers wrapping the core methods with `FileStream`/`StreamReader`.
4. `BaseType.ResetLastTopNode()` called before and after deserialization, to clear the
   `[ThreadStatic]` `LastTopNode` field before constructing a new tree (see
   [thread-safety.md](thread-safety.md)).

| Serializer | Library | Polymorphism mechanism |
|---|---|---|
| `SdcSerializer<T>` | `System.Xml.Serialization.XmlSerializer` | ~150 `[XmlInclude]` attributes on `BaseType` |
| `SdcSerializerJson<T>` | `Newtonsoft.Json` | `TypeNameHandling.All` writes `"$type"` |
| `SdcSerializerBson<T>` | `Newtonsoft.Json` + `Newtonsoft.Json.Bson` | same `TypeNameHandling.All`, via the shared cached `JsonSerializer` |
| `SdcSerializerMsgPack<T>` | `Newtonsoft.Msgpack` (`MsgPack.Cli`/`MsgPack.Serialization`) | native CLR reflection only — no type discriminators (see MsgPack status above) |

## Advanced JSON settings available for future tuning

The following `Newtonsoft.Json.JsonSerializerSettings` are exposed as Xsd2Code++ generator options
and may be worth enabling for the SDC OM in the future (tracked in [roadmap.md](roadmap.md)):
`DateFormatHandling`, `DateFormatString`, `DateParseHandling`, `DateTimeZoneHandling`,
`DefaultValueHandling` (`Ignore` would omit `[DefaultValue]`-matching properties and shrink
payload size), `FloatFormatHandling`, `FloatParseHandling`, `MissingMemberHandling`,
`NullValueHandling` (`Ignore` would skip null properties and shrink payload size),
`StringEscapeHandling`.
